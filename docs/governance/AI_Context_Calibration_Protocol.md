\# Smart Business AI Context Calibration Protocol

\*\*Artifact Type:\*\* Governance Protocol

\*\*Version:\*\* 1.0

\*\*Status:\*\* Approved

\*\*Repository Location:\*\*

docs/governance/AI\_Context\_Calibration\_Protocol.md

\---

\# Purpose

This protocol defines the standard procedure for validating that an AI-assisted development environment correctly understands and respects the approved Smart Business governance framework before implementation begins.

The objective is to replace assumptions with verified operational understanding.

The protocol evaluates AI behaviour.

It does not evaluate application functionality.

\---

\# Scope

This protocol shall be completed before the first governed Build Mode execution on any new AI development platform or implementation environment.

Examples include:

\- Lovable  
\- Claude Code  
\- Cursor  
\- Windsurf  
\- Future approved AI development platforms

Mission Control may also authorize this protocol after significant changes to:

\- Workspace Knowledge  
\- Project Knowledge  
\- Repository governance  
\- AI platform behaviour  
\- AI model versions

\---

\# Principles

The calibration shall:

\- remain in planning mode whenever possible,  
\- avoid application implementation,  
\- avoid code generation,  
\- avoid modifying repository content,  
\- evaluate observed behaviour rather than assumptions.

All conclusions shall be evidence-based.

\---

\# Calibration Categories

The following areas shall be evaluated.

\#\# 1\. Product Identity

Verify understanding of:

\- Product identity  
\- Target users  
\- Product purpose

\---

\#\# 2\. Governance Awareness

Verify understanding of:

\- Founder authority  
\- Mission Control authority  
\- Product Truth  
\- Human-in-the-Loop governance

\---

\#\# 3\. Repository Awareness

Verify understanding of:

\- README.md  
\- AGENTS.md  
\- Repository documentation

Classify repository awareness as:

\- Verified  
\- Partially Verified  
\- Not Verified

Do not assume undocumented platform behaviour.

\---

\#\# 4\. Implementation Boundaries

Verify understanding of:

\- authorized implementation scope,  
\- prohibited implementation,  
\- implementation stop conditions.

\---

\#\# 5\. Route Awareness

Verify approved routes.

Confirm deprecated routes are not introduced.

\---

\#\# 6\. Brand & User Experience

Verify understanding of:

\- Brand identity  
\- User experience philosophy  
\- Design principles

\---

\#\# 7\. Governance Conflict Handling

Present an intentionally conflicting request.

Verify that the AI:

\- refuses unauthorized implementation,  
\- explains the conflict,  
\- requests clarification,  
\- preserves Product Truth.

\---

\#\# 8\. AI Context Hierarchy

Verify that the AI correctly distinguishes between:

\- Organization context  
\- Project context  
\- Repository documentation  
\- Temporary conversation context

\---

\#\# 9\. Implementation Readiness

Ask the AI to summarize:

\- what it is authorized to implement,  
\- what remains prohibited,  
\- where implementation must stop.

\---

\# Assessment

Each category shall receive one of:

\- PASS  
\- PASS WITH OBSERVATIONS  
\- FAIL

\---

\# Evidence

Record:

\- Calibration questions  
\- Expected behaviour  
\- AI responses  
\- Mission observations

Do not rely on memory.

\---

\# Final Recommendation

Provide one of:

\- Ready for Implementation  
\- Ready with Observations  
\- Not Ready

Support the recommendation with evidence.

\---

\# Governance Decision

Mission Control shall record:

\- PASS  
\- PASS WITH OBSERVATIONS  
\- FAIL

Only a PASS or PASS WITH OBSERVATIONS may authorize implementation.
